## ODR Safety Rationale (from cross-AI review)

The review raised concern about ODR violations from including model headers in test TUs.
After source analysis, this is a FALSE POSITIVE for this codebase:

1. All model headers (clip.hpp, t5.hpp, vae.hpp, flux.hpp) are FULLY HEADER-ONLY.
   All member functions are defined within struct/class bodies = implicitly inline.
2. ggml_extend.hpp uses `static inline` for free functions and inline struct methods.
3. model.h uses `static inline` for all helper functions.
4. Including these headers from test TUs is safe -- the linker deduplicates
   identical inline definitions across wan-api.cpp and test TUs.

CONCLUSION: Tests CAN and SHOULD directly `#include "clip.hpp"` etc. to access
Runner class constructors. The wan-cpp library provides the compiled object code,
and test TUs provide their own inline copies -- standard C++ inline function behavior.

## Verified Enum Values

VERSION_FLEX_2 is CORRECT (not a typo for FLUX_2). Verified in src/model.h line 42.
VERSION_FLUX2 is a separate enum value at line 49. Both exist as distinct versions.

## Empty String2TensorStorage Safety

Empty storage IS SAFE for all 4 model constructors. Model constructors call
`model.init(params_ctx, tensor_storage_map, prefix)` which calls `init_params()`
and `init_blocks()` -- these create tensors in params_ctx regardless of storage map
contents. The storage map is only used for type inference (get_type() defaults to
GGML_TYPE_F16 when tensor name not found). alloc_params_buffer() succeeds because
tensors exist in params_ctx.

## String2TensorStorage Type Correction

String2TensorStorage is `OrderedMap<std::string, TensorStorage>` (from model.h line 290),
NOT `std::map`. OrderedMap is defined in src/ordered_map.hpp. Default construction
via `String2TensorStorage{}` creates an empty ordered map -- this is safe.

```cmake
# wan_add_test(target_name source_file)
# Creates a test executable linked against wan-cpp with proper include paths.
# Tests access model headers (clip.hpp, t5.hpp, etc.) through wan-cpp's PUBLIC
# include directories -- this is safe because all model headers are fully
# header-only with implicitly inline member functions.
function(wan_add_test target_name source_file)
    add_executable(${target_name} ${source_file})
    target_link_libraries(${target_name} PRIVATE wan-cpp)
    target_include_directories(${target_name} PRIVATE
        ${CMAKE_SOURCE_DIR}/src          # Model headers (clip.hpp, t5.hpp, etc.)
        ${CMAKE_SOURCE_DIR}/include      # Public API headers
        ${CMAKE_CURRENT_SOURCE_DIR}      # Test headers (test_framework.hpp, etc.)
    )
    target_compile_features(${target_name} PRIVATE cxx_std_17)
    add_test(NAME ${target_name} COMMAND ${target_name})
endfunction()

wan_add_test(test_factory     test_factory.cpp)
wan_add_test(test_clip        test_clip.cpp)
wan_add_test(test_t5          test_t5.cpp)
wan_add_test(test_vae         test_vae.cpp)
wan_add_test(test_transformer test_transformer.cpp)
```

Note: `target_link_libraries(... PRIVATE wan-cpp)` provides all the GGML, ggml-backend, spdlog symbols. The `${CMAKE_SOURCE_DIR}/src` include directory is needed because wan-cpp declares `src/` as a PUBLIC include directory already, but adding it explicitly ensures test files can `#include "clip.hpp"` etc. directly.

**5. Create tests/cpp/test_helpers.hpp** (per D-12, D-13):

Shared test utilities:

- `enum class T5Version { STANDARD_T5, UMT5 };` -- custom enum since T5Runner uses `bool is_umt5` (see Pitfall 3 in RESEARCH.md). Factory creator lambdas convert: `T5Version::STANDARD_T5` -> `is_umt5=false`, `T5Version::UMT5` -> `is_umt5=true`. (Addresses review concern about explicit T5Version conversion.)

- `struct BackendRAII` (addresses review concern about BackendRAII specification):
  ```cpp
  struct BackendRAII {
      ggml_backend_t backend;

      explicit BackendRAII(ggml_backend_t b) : backend(b) {}
      ~BackendRAII() {
          if (backend) {
              ggml_backend_free(backend);  // Safe for CPU backend
          }
      }

      // Non-copyable, non-assignable -- each test owns its backend
      BackendRAII(const BackendRAII&) = delete;
      BackendRAII& operator=(const BackendRAII&) = delete;
  };
  ```

  Lifetime management rationale: GGMLRunner's destructor does NOT free its
  `runtime_backend` (the backend passed to its constructor). It only frees a
  separate `params_backend` if offload_params_to_cpu created one. Therefore,
  the test code MUST free the backend explicitly. BackendRAII handles this.
  The correct destruction order is: Runner destroyed first (freeing internal
  buffers), then BackendRAII destroyed (freeing the backend). This happens
  naturally when BackendRAII is declared before the Runner in the same scope,
  since C++ destroys locals in reverse order.

- Include `#pragma once`, `"ggml-backend.h"`
