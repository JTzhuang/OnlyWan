<task type="auto">
  <name>Task 1: OP-02 — RoPE 位置编码 GPU 化</name>
  <files>src/rope.hpp, src/wan.hpp</files>
  <action>
  **目标：** 将 `gen_wan_pe` 的计算从 CPU std::vector 改为 GGML graph node，让 PE 在 GPU 上直接计算。

  **方案：** 在 GGML 计算图中构建 PE 生成逻辑，而非在 CPU 上预计算后上传。

  在 `src/rope.hpp` 中：
  1. 添加新函数 `gen_wan_pe_tensor()`，接受 `ggml_context*` 和维度参数，返回 `ggml_tensor*`
  2. 函数内部使用 GGML ops 构建 PE 计算：
     - 使用 `ggml_arange` 生成 t/h/w 索引序列
     - 使用 `ggml_mul` / `ggml_div` 计算频率
     - 使用 `ggml_sin` / `ggml_cos` 计算 sin/cos 编码
     - 使用 `ggml_concat` 组合各维度
  3. 如果 GGML 缺少某些 ops（如 arange），退回到 CPU 预计算但使用 `ggml_new_tensor` + `memcpy` 的方式，将数据作为 graph 的常量输入（而非每步重新计算）

  **备选方案（如果 GGML ops 不足）：**
  - 在 WanRunner 的 `build_graph` 中，将 PE 作为缓存 tensor 处理
  - 首次调用时 CPU 计算 PE 并缓存到 `cache_tensor_map`
  - 后续调用直接从缓存读取，避免重复计算和上传

  在 `src/wan.hpp` 的 WanRunner 中：
  1. 在 `build_graph` 或 `compute` 中，检查 PE 是否已缓存
  2. 如果未缓存，调用 `gen_wan_pe` 计算并存入缓存
  3. 如果已缓存，直接使用缓存的 PE tensor

  注意：优先使用备选方案（PE 缓存），因为 GGML 的 op 支持可能不完整。这仍然消除了每步重复的 CPU 计算和 CPU->GPU 传输。

<task type="auto">
  <name>Task 2: FUS-02 — Linear + GELU 算子融合</name>
  <files>src/ggml_extend.hpp, src/wan.hpp</files>
  <action>
  **目标：** 融合 FFN 块中的 Linear + GELU 操作，减少中间内存分配。

  **方案：** GGML 层面的融合 — 利用 `ggml_gelu_inplace` 直接在 Linear 输出上操作，避免额外的内存分配。

  在 `src/ggml_extend.hpp` 中：
  1. 添加辅助函数 `ggml_ext_linear_gelu()`：
     ```cpp
     __STATIC_INLINE__ struct ggml_tensor* ggml_ext_linear_gelu(
         struct ggml_context* ctx,
         struct ggml_tensor* x,
         struct ggml_tensor* weight,
         struct ggml_tensor* bias) {
         // mul_mat + bias + gelu_inplace 链式调用
         auto y = ggml_mul_mat(ctx, weight, x);
         if (bias) y = ggml_add(ctx, y, bias);
         y = ggml_gelu_inplace(ctx, y);
         return y;
     }
     ```
  2. 这确保 GGML 后端可以识别 inplace 模式并优化内存

  在 `src/wan.hpp` 的 `WanAttentionBlock::forward` 中（第 1574-1576 行）：
  1. 将原来的三步调用：
     ```cpp
     y = ffn_0->forward(ctx, y);
     y = ggml_ext_gelu(ctx->ggml_ctx, y, true);
     ```
     替换为：
     ```cpp
     // 直接获取 ffn_0 的 weight 和 bias，使用融合路径
     auto ffn_0_w = ffn_0->get_weight();
     auto ffn_0_b = ffn_0->get_bias();
     y = ggml_ext_linear_gelu(ctx->ggml_ctx, y, ffn_0_w, ffn_0_b);
     ```
  2. 如果 Linear 类没有 `get_weight()`/`get_bias()` 方法，保持原有调用方式但确保 `ggml_ext_gelu` 使用 `inplace=true`（当前已经是 true）

  **保守方案（如果 Linear 接口不便修改）：**
  - 确认当前 `ggml_ext_gelu(ctx->ggml_ctx, y, true)` 的 inplace 模式已经生效
  - 验证 GGML 后端是否正确处理 inplace gelu
  - 如果已经是 inplace，则 FUS-02 的收益主要来自 CUDA graph 捕获时的 kernel 融合（由 CG-02 启用）
  - 在这种情况下，添加注释说明融合依赖 CUDA graph 的 kernel 合并

  注意：不要引入自定义 CUDA kernel。利用 GGML 现有的 inplace 操作和 CUDA graph 的自动优化。
