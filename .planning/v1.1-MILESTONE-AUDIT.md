## Critical Gaps (Blockers)

### Gap 1: PERF-01 — vocab dir never set, T2V/I2V silently broken

**Affected requirements:** PERF-01, ENCODER-01, ENCODER-02, API-03, API-04

`wan_vocab_set_dir()` is implemented in `src/vocab/vocab.cpp` and declared in `src/vocab/vocab.h` (internal header). It is:
- Never called in `wan-api.cpp`, `wan-cli/main.cpp`, or `wan-convert/main.cpp`
- Not exposed in the public `wan.h` header
- Not called during `wan_load_model`

At runtime with `WAN_EMBED_VOCAB=OFF` (the default build), `g_vocab_dir` is always `""`. All six `load_*` functions return empty strings. `T5Embedder::tokenize()` produces an empty token list. `wan_generate_video_t2v_ex` checks `if (tokens.empty())` and returns `WAN_ERROR_INVALID_ARGUMENT` — every T2V and I2V call fails silently.

**Fix required:**
1. Expose `wan_set_vocab_dir(const char* dir)` in `wan.h`
2. Call it in `wan-cli/main.cpp` (add `--vocab-dir` flag or use model directory as default)
3. Or: call `wan_vocab_set_dir` inside `wan_load_model` using the model file's parent directory as default

### Gap 2: SAFE-03 — vae/t5/clip sub-model GGUFs have no load path

**Affected requirements:** SAFE-03

`wan-convert` correctly produces GGUF files for `--type vae`, `--type t5`, `--type clip` with proper metadata. However, `wan_load_model` has no mechanism to load a single-submodel GGUF — it always constructs a full `WanRunner` from `model.diffusion_model.*` tensors. A vae-only or t5-only GGUF will load zero DiT tensors and produce an empty runner.

**Fix required (choose one):**
1. Add a multi-file `wan_load_model` variant that accepts separate paths for DiT, VAE, T5, CLIP
2. Or: document that `wan-convert` vae/t5/clip types are for future use and only `dit-*` types produce files loadable by the current `wan_load_model`
